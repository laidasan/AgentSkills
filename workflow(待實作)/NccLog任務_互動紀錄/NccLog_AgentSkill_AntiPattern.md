# NccLog 埋設工作流程 — Anti-Pattern 與注意事項

> 本文件記錄應避免的行為模式與邊界情況，補充 Pattern 文件的正面描述。
> 來源：多次 NccLog 實作對話中觀察到的風險點。

---

## 一、流程層面的 Anti-Pattern

### AP-1：跳過分析直接進實作規劃

**錯誤行為**：使用者說「幫我實作 XX 頁面的 NccLog」，Agent 直接進入 PlanMode 開始寫步驟。

**為什麼有害**：
- 沒有對照已完成頁面，可能遺漏關鍵差異
- 沒有浮現議題，會在實作中途才發現需要重新設計
- 使用者無法在前期介入決策

**正確做法**：先完成 Phase 2（差異分析），確保所有差距項目已列出，再進入規劃。

---

### AP-2：沒有浮現議題就直接輸出規劃

**錯誤行為**：差異分析完成後，Agent 認為「都很明確」，直接輸出實作計畫。

**為什麼有害**：
- 「看似明確」的項目中可能藏有隱含假設（如本次的 garId/pr 是否存在、serialNumber 如何計算）
- 使用者可能有不同的設計偏好，但沒有機會表達

**正確做法**：即使看似明確，也要至少列出 2-3 個需確認的點。如果真的完全明確，說明為什麼明確（例如「因為 API 規格已確認不回傳 garId」）。

---

### AP-3：規劃審視時只問「可以嗎？」而不主動找問題

**錯誤行為**：輸出規劃後，Agent 說「規劃如上，可以開始實作嗎？」

**為什麼有害**：
- 錯失發現遺漏的機會（如本次的 ResumeLinkFactory RC 用途、input 欄位問題）
- 使用者期望 Agent 在審視階段**主動發現問題**

**正確做法**：Agent 主動重新審視規劃，提出自己發現的議題、疑慮、改善建議。使用者會在此階段給回饋。

---

### AP-4：使用者說「開始實作」前就動手改代碼

**錯誤行為**：規劃完成後，Agent 自行判斷「應該沒問題了」就開始修改檔案。

**為什麼有害**：
- 使用者可能還沒看完規劃
- 使用者可能想調整某些步驟的順序或內容

**正確做法**：等待使用者明確表達「開始實作」、「沒有其他問題」等確認訊號。

---

## 二、分析與討論層面的 Anti-Pattern

### AP-5：直接附和使用者的 Solution 而不驗證

**錯誤行為**：使用者提出「我覺得 pageId 就是 jobNo」，Agent 回「好的，那就用 pageId 取 jobNo」。

**為什麼有害**：
- 使用者的直覺可能基於不完整的資訊
- 本次案例中 pageUid 其實是隨機 UUID，和 jobNo 完全無關

**正確做法**：
1. 先去**探索代碼驗證**使用者的假設
2. 如果正確，回報驗證結果並採用
3. 如果不正確，用**具體證據**說明（代碼片段、資料結構），而非只說「不對」

---

### AP-6：否定使用者的 Solution 時只說「不行」而不舉例

**錯誤行為**：使用者提出 serialNumber 用 `index + resumeCardIndex + 1`，Agent 回「這個不正確，因為每個職缺履歷數不同」。

**為什麼有害**：
- 抽象的解釋不夠有說服力
- 使用者可能覺得 Agent 沒有認真考慮

**正確做法**：用**具體的數字範例**展示問題。例如：

> Job 0 (3筆): 0+0+1=1, 0+1+1=2, 0+2+1=3 ✓
> Job 1 (2筆): 1+0+1=2, 1+1+1=3 ✗（應該是 4, 5）

表格形式更直觀，讓使用者自己看出問題。

---

### AP-7：議題討論時替使用者做決定

**錯誤行為**：Agent 說「garId/pr 固定帶空字串，我決定不做存儲邏輯」。

**為什麼有害**：
- 使用者可能有不同考量（如未來擴展性）
- 跳過確認會讓使用者失去掌控感

**正確做法**：列出選項和影響，讓使用者決定。例如：

> 這裡要確認：`/resume/key-jobs/recommend` 的 API response 到底有沒有回傳 garId / pr？
> 如果沒有，那 customNccLogExt 的 Gar_id 和 pr 就確定帶空字串，不需要存儲邏輯。

---

## 三、實作規劃層面的 Anti-Pattern

### AP-8：規劃文件只列步驟，不記錄設計決策的脈絡

**錯誤行為**：實作計畫只寫「Step 3: 新增 getCustomNccLogExt 方法，回傳空字串的 Gar_id 和 pr」。

**為什麼有害**：
- 新 session 或其他開發者看到會問「為什麼是空字串？」
- 無法追溯決策原因

**正確做法**：在規劃文件中包含「已確認的設計決策」段落，記錄：
- 決策內容
- 決策原因（如「API 不回傳 garId/pr，經使用者確認」）

---

### AP-9：只列「要改什麼」，不列「不需要改什麼」

**錯誤行為**：規劃文件只包含修改步驟，不提「不需修改的部分」。

**為什麼有害**：
- 使用者可能擔心是不是遺漏了（如 sendFeedbackNccLog 是否需要改？）
- 後續 review 時無法判斷是「有意不改」還是「忘記了」

**正確做法**：明確列出不需修改的部分及原因。例如：

> | sendFeedbackNccLog | pr/Gar_id 已正確為空字串，jobNo 已從 emit 參數取得 |

---

### AP-10：規劃中的代碼範例缺少上下文

**錯誤行為**：只貼修改後的代碼片段，不標注檔案路徑和行號。

**為什麼有害**：
- 實作時不知道改哪裡
- 多個檔案有類似方法名稱時容易改錯

**正確做法**：每個步驟標明**檔案路徑**和**行號**，代碼範例包含足夠的上下文（前後幾行）。

---

## 四、邊界情況

### BC-1：兩個頁面的資料結構差異比預期大

**情境**：參照頁面用 Vuex state 存全域值（如 jobNo），目標頁面是多職缺結構，同一個值在不同 card 上不同。

**處理方式**：
- 不能直接照搬 getter 模式
- 需要改用方法（method），從 resumeCardUid 動態取得
- 在議題討論階段就要識別出這個差異

### BC-2：使用者提出的公式看似合理但有隱含前提

**情境**：使用者建議用 AiResumeList 的 `(page-1)*perPage+(index+1)` 公式，因為在參照頁面確實有效。

**處理方式**：
- 分析公式成立的**前提條件**（每頁筆數固定）
- 確認目標頁面是否滿足該前提
- 若不滿足，用具體例子說明斷裂點

### BC-3：「已完成」的部分恰好是正確值

**情境**：sendFeedbackNccLog 中 hardcode 的 `pr: ''` 和 `Gar_id: ''` 恰好是此頁面的正確值。

**處理方式**：
- 明確記錄「不需修改，因為現有值恰好正確」
- 而非默默跳過不提（會被誤認為遺漏）
