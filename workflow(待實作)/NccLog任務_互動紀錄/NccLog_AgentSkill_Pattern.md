# NccLog 埋設工作流程 — Pattern 文件

> 本文件描述「NccLog 埋設」類型任務的標準工作流程 Pattern，供 AgentSkill 設計參考。
> 此 Pattern 從多次 NccLog 實作對話中歸納而來，適用於「參照已完成頁面，為另一個頁面補齊 NccLog」的場景。

---

## 一、工作流程總覽

```
Phase 1: 建立脈絡
    ↓
Phase 2: 分析差異
    ↓
Phase 3: 議題討論（可多輪）
    ↓
Phase 4: 實作規劃（輸出 md 文件）
    ↓
Phase 5: 規劃審視（可多輪）
    ↓
Phase 6: 實作執行
```

每個 Phase 都有明確的**進入條件**和**產出物**，不可跳過。

---

## 二、各 Phase 詳細定義

### Phase 1：建立脈絡

**目的**：讓 Agent 和使用者對齊背景知識，確保雙方在同一個認知基礎上。

**行為規範**：
- 閱讀使用者指定的參考文件（分析文件、脈絡總整理、Excel 規格等）
- 閱讀已完成實作的相關代碼（作為參照基準）
- **不主動提出方案**，先確認自己理解正確

**產出物**：Agent 確認已理解脈絡，向使用者表達準備就緒

---

### Phase 2：分析差異

**目的**：對照「已完成的實作」與「待實作的目標」，系統性列出差距。

**行為規範**：
- 以**表格形式**呈現差異對照（已完成 vs 待完成）
- 明確標示每個差距項目的**狀態**（已有 / 未設定 / 不適用）
- 區分「可直接沿用的模式」和「因頁面特性不同需要調整的部分」

**產出物**：差異分析表，列出所有需補齊的項目

---

### Phase 3：議題討論

**目的**：針對分析中發現的不確定點，逐一與使用者討論並確認設計決策。

**行為規範**：
- 將不確定點整理為**編號議題**（議題 1、議題 2...），方便逐一回應
- 每個議題要包含：**問題描述**、**影響範圍**、**可能的選項或方向**
- **不替使用者做決定**，而是提供足夠資訊讓使用者判斷
- 使用者回覆後，確認理解並記錄決策結論
- 若使用者回覆引發新的議題，繼續討論（可多輪）

**產出物**：所有議題的決策結論

**品質檢查**：
- [ ] 所有不確定點都已討論
- [ ] 每個決策都有明確結論
- [ ] 沒有遺留模糊的假設

---

### Phase 4：實作規劃

**目的**：將分析結果和設計決策轉化為可執行的步驟化實作計畫。

**行為規範**：
- 進入 PlanMode 進行規劃
- 規劃內容需包含：
  - **Context**：為什麼要做這件事、關鍵差異
  - **已確認的設計決策**：Phase 3 的結論整理
  - **修改步驟**：每步標明檔案、方法、具體改動，附代碼範例
  - **不需修改的部分**：明確列出並說明原因（避免遺漏的誤解）
  - **修改檔案清單**：總覽
  - **驗證方式**：如何確認實作正確
- 輸出為**獨立的 md 文件**（命名格式：`NccLog_實作計畫_{類別名稱}.md`）

**產出物**：實作計畫 md 文件

---

### Phase 5：規劃審視

**目的**：在實作前重新檢視規劃，發現遺漏或可改善的點。

**行為規範**：
- Agent 主動審視規劃，提出可能的**議題、問題、想法**
- 議題格式同 Phase 3（編號、描述、影響、選項）
- 使用者確認後，將結論更新回實作計畫 md 文件
- **不在使用者說「開始實作」之前動手寫 code**

**產出物**：更新後的實作計畫（如有修改）

**品質檢查**：
- [ ] 使用者明確說「沒有其他問題」或「開始實作」
- [ ] 所有設計決策已記錄在實作計畫中

---

### Phase 6：實作執行

**目的**：依據實作計畫逐步修改代碼。

**行為規範**：
- 嚴格按照實作計畫的步驟順序執行
- 每個步驟完成後可用 TaskUpdate 追蹤進度
- 完成後輸出修改摘要表（檔案 + 改動內容）

**產出物**：完成的代碼修改 + 修改摘要

---

## 三、使用者偏好

### 溝通風格
- 偏好**先分析再實作**，不跳步驟
- 重視**議題浮現與討論**，Agent 不該直接給結論跳過討論
- 設計決策需要有**脈絡紀錄**（為什麼這樣決定、考慮過什麼替代方案）
- 規劃文件要能**獨立閱讀**，新 session 也能看懂

### 輸出格式
- 差異分析用**表格**呈現
- 議題用**編號**方便逐一回應
- 實作計畫輸出為**獨立 md 文件**
- 代碼範例要標注**檔案路徑和行號**

### 決策風格
- 使用者會主動提出自己的 Solution 想法，Agent 要**評估可行性**而非直接附和
- 若使用者的 Solution 有問題，Agent 要**用具體例子說明為什麼不行**（如本次 serialNumber 的 index 計算問題）
- 評估後若確實不可行，提供正確方案並說明原因

## 四、跨任務的共通模式

以下模式在多次 NccLog 實作中反覆出現：

### 4.1 「參照已完成頁面」模式
- 先徹底理解已完成頁面的實作方式（代碼、資料流、helper 函式）
- 再評估目標頁面的**結構差異**（資料來源、操作類型、uid 格式等）
- 判斷哪些可以**直接沿用**、哪些需要**調整**

### 4.2 「customNccLogExt 策略」模式
- 先確認 ext 欄位的**資料來源**（API 回傳 / Vuex state / resumeCard 屬性）
- 區分「固定值」vs「動態值」
- 固定值 → getter；依操作而異 → 方法

### 4.3 「議題 → 決策 → 記錄」循環
- 每個不確定點都經過：浮現議題 → 討論選項 → 使用者決策 → 記錄到文件
- 決策記錄不只是結論，還包含**為什麼排除其他選項**
