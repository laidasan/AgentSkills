# 討論紀錄 02 — Workflow 機制設計

日期：2026-07-01

## 討論背景
在確認 workflow 結構性議題（discussion-01）後，進入 workflow.md 撰寫階段前，針對 optional 流程和 Human-in-loop 的具體執行機制進行設計討論。

## 議題與決策

### 1. Optional 步驟的處理方式
**議題**：如何讓 Claude 知道某個步驟是 optional 的？

**決策**：分兩類處理——
- **功能性步驟**（步驟一、二、四）：用條件自動判斷，不需要問使用者。例如「有既有 codebase 且使用者未提供規格文件 → 執行步驟一」。
- **Human-in-loop 確認步驟**（步驟三、五、九）：由使用者在初始化階段決定是否保留。

### 2. 初始化階段的兩輪設計
**議題**：A 方案（流程開頭一次性收集）vs B 方案（到了再問）？

**決策：採 A 方案，且拆為兩輪。**
- **第一輪**：收集使用者提供了什麼輸入物（codebase、規格、架構文件），由 Claude 自動決定功能性步驟的 skip/execute。
- **第二輪**：列出 optional 的 Human-in-loop 步驟，由使用者決定哪些要保留。

**理由**：
- A 方案讓 Claude 在流程開始時就建立完整執行路徑，執行連貫性更好。
- 拆兩輪是因為使用者不需要理解整個流程，只需回答「我有什麼」和「哪些確認步驟要保留」。

### 3. Human-in-loop 的執行機制
**議題**：Claude 在 Human-in-loop 步驟具體要怎麼做？

**決策**：定義為通用規則，所有 Human-in-loop 步驟共用——
1. 呈現產出物。
2. 詢問是否正確、是否需要修正。
3. 停下來等待回應，不可自行往下執行。
4. 若使用者提出修正 → 更新後重新呈現 → 再次詢問，循環直到確認通過。

### 4. 步驟一、二的具體觸發條件
**議題**：步驟一二什麼時候該執行？

**決策**：「沒有就補，有就跳」——
- 使用者有既有 codebase 但沒提供規格文件 → 執行步驟一（分析 codebase 補規格）。
- 使用者有既有 codebase 但沒提供架構文件 → 執行步驟二（分析 codebase 補架構）。
- 已有對應文件 → 跳過。
