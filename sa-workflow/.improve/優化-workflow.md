# SA Workflow 優化計畫

## 優化方式

透過實際執行 SA Workflow 來驗證流程設計，以迭代方式持續改進。

### 執行循環

1. **Run Workflow** — 使用測試規格文件，完整走過 SA Workflow 的各步驟
2. **記錄問題** — 過程中遇到的問題、模糊地帶、不順暢之處，記錄於改善建議中
3. **產出改善建議** — 針對問題提出具體的修正方向與建議
4. **更新 Workflow** — 確認改善方向後，修正 workflow.md 或相關 rules
5. **下一輪驗證** — 再次執行，驗證修正效果

### 測試素材

- `.test-doc/0_整體流程全貌.md` — B Agent 整體流程（主幹 + 4 個子流程）
- `.test-doc/11_互動回饋模組(r).md` — 互動回饋模組的詳細規格（子流程 C 的完整規格）

### 產出文件

每輪執行後產出：

| 文件 | 位置 | 用途 |
|------|------|------|
| 改善建議 | `.improve/改善建議-round-{N}.md` | 該輪執行中發現的問題、交流紀錄、改善方向 |
| Session Recap | `.recap/RECAP.md` | 更新當前進度，供下次 session 延續 |

## 量化評分方向（尚未執行，方向記錄）

以下為未來可能啟用的評分維度，基於 SA Workflow 的特性設計。目前僅作為方向記錄，待實際跑過數輪後再決定是否啟用、如何量化。

### 維度一：步驟產出物品質

評估每個步驟的產出物是否符合預期。

- 規格分析（步驟一）：是否完整描述既有系統的功能與行為
- 架構分析（步驟二）：ClassDiagram / SequenceDiagram 是否準確反映 codebase 結構
- 差異比對（步驟四）：是否精確識別新舊規格的差異，無遺漏、無誤判
- SA 文件（步驟六）：功能分類是否合理、依賴關係是否正確
- Design 文件（步驟八）：架構設計是否可實作、ClassDiagram / SequenceDiagram 是否一致
- Task 清單（步驟十）：拆解粒度是否適當、是否可獨立執行

### 維度二：流程銜接正確性

評估步驟之間的邏輯銜接是否正確。

- 初始化階段的條件判斷是否正確決定執行路徑
- 跳過步驟時，後續步驟是否正確處理缺少的輸入
- Human-in-loop 修正後，是否正確影響後續步驟的產出

### 維度三：Human-in-loop 互動品質

評估與使用者互動的品質。

- 是否在正確的時機暫停等待使用者確認
- 呈現產出物時，是否讓使用者容易判斷正確性
- 使用者提出修正時，是否準確理解並反映修正意圖
- 是否避免不必要的來回（一次就問清楚）

### 維度四：規範遵循度

評估執行過程是否遵循 `.claude/rules/` 下的各項規範。

- Diagram 類：flowchart / sequence diagram 是否符合繪製原則
- Analysis 類：Vue SFC 分析是否以 class 視角呈現
- Development 類：設計是否符合 SOLID、編碼風格等原則

### 啟用時機

當以下條件滿足時，考慮啟用量化評分：

- 已完成至少 2 輪完整的 workflow 執行
- 改善建議中出現重複的問題模式，需要量化追蹤改善進度
- 評分維度經過實際驗證，確認可操作且有區分度
