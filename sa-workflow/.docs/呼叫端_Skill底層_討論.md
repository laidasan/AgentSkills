# 呼叫端（Caller）與 Skill 底層的職責劃分 — 完整對話紀錄

> 日期：2026-07-07
> 背景：討論 analyst-system SKILL.md 的架構重構方向

---

## 使用者開場

看一下 `skills/analyst-system/SKILL.md`。我有些想法，想和你一起討論，集思廣益。

---

## 使用者提出想法

目前，在 Skill 文件前會先設定需要遵循的規則、輸出檔案的結構。我的想法是，將這些從 Skill.md 移除，也就是從 Skill 規範中移除。我認為這份 analyst-system skill 是一個設計好的 workflow，不應該依賴相關的法則和輸出檔案結構，至多能夠設計樣板，來設計產出。

取而代之，讓呼叫「analyst-system」的 AgentSkill、Command（claude/commands）的工具，定義額外應該要遵守的 rules 和結構。如此，「analyst-system」會是一個獨立且相對乾淨的流程，也能夠彈性的讓外部進行包裝與使用。

你覺得這個想法如何？有什麼其他想法或建議嗎？

---

## Claude 回應

這個方向我認為是合理的，本質上就是把 analyst-system 當作一個「純流程引擎」，rules 和輸出結構變成外部注入的依賴，而不是內建綁定。這跟 Dependency Inversion 的思路一致——workflow 依賴抽象（「需要有 rules」「需要有輸出結構」），不依賴具體的規範內容。

不過有幾個點值得先釐清：

**1. 注入機制怎麼設計？**

目前 SKILL.md 裡直接用 `@rules/...` 引用，移除後，caller 要用什麼方式把 rules 傳進來？幾種可能：
- Caller 的 command/skill 裡用 `@` 引用 rules，analyst-system 只聲明「我需要哪些類別的規範」（如：diagram 規範、開發規範、分析規範）
- 或是 analyst-system 定義一組「slot」，caller 填入對應的規範路徑

這會影響 skill 本身怎麼描述「我預期外部提供什麼」。

**2. 輸出目錄結構移除後，workflow 步驟裡的產出路徑怎麼處理？**

目前每個步驟都寫死了路徑（如 `.sa/design/`、`.tasks/`）。移除後，步驟描述會變成「產出 Design 文件到指定目錄」這種抽象描述，具體路徑由 caller 定義。但這會讓 workflow 本身讀起來比較不具體——這個取捨你可以接受嗎？

**3. 「至多能夠設計樣板」這個邊界怎麼拿捏？**

樣板定義的是產出物的「結構」（哪些 section、什麼格式），而 rules 定義的是「怎麼寫內容」（diagram 怎麼畫、code 怎麼組織）。這兩者的分界在大部分情況下很清楚，但像 Task 文件規範同時定義了結構（必要 section）和內容規則（命名格式、metadata 欄位），這種情況你會怎麼歸類？

## Claude 回應

理解了，整體架構就是：

- **analyst-system**：純 workflow + 輸出樣板（格式）
- **executing-sa（command）**：注入 rules 路徑、輸出目錄結構、額外 context
- **使用者**：透過 command 參數補充自然語言 context 或額外規範

這個分層很清楚，職責劃分也合理。

我有一個延伸的想法：

**analyst-system 是否需要聲明「我預期外部提供什麼」？**

目前的設計是 caller 自行負責注入，analyst-system 本身不檢查。但如果 analyst-system 在開頭有一段類似「前置條件」的聲明——例如「執行此 workflow 前，caller 應提供：diagram 繪製規範、開發規範、輸出目錄定義」——有兩個好處：

1. 寫新的 caller（比如未來有另一個 command 也要呼叫 analyst-system）時，知道該準備什麼
2. 當某個 caller 漏了某類規範，執行時至少能意識到缺少了什麼，而不是默默用 LLM 的預設行為產出

不過這也可能過度設計——如果 caller 數量少且你自己控制，靠慣例就夠了。你怎麼看，需要這層聲明嗎？

## Claude 從多角度挑戰

你的論點邏輯上是自洽的——analyst-system 作為純流程，不應該反過來約束 caller。要加文件而不是加機制，這個判斷我同意。

不過讓我從幾個不同角度挑戰一下：

**從實際執行面看：LLM 沒有「編譯器」幫你擋**

傳統的 DI 框架，少注入一個依賴會直接報錯。但這裡的 caller 是另一份 markdown，注入的是自然語言 context，執行者是 LLM。如果 caller 漏了 diagram 規範，LLM 不會報錯，它會用自己的判斷畫出一個「看起來合理但不符合你預期」的 diagram。這個問題不會在執行時被發現，只會在你 review 產出物時才察覺。

所以問題不是「要不要在 analyst-system 裡加前置條件」，而是——你有沒有其他機制能在這種 silent failure 發生時 catch 到？

幾個可能的方向：
- 使用文件中列出 checklist（你已經提到的）
- executing-sa command 本身有自己的 self-check（這是 caller 的責任，不違反你的原則）
- 或者你就接受這個風險，因為 caller 數量少，實際上不太會漏

**從「樣板留在 skill 裡」這個決定延伸**

你把樣板（格式）留在 analyst-system，rules（內容規範）放在 caller。但有些東西會落在灰色地帶——比如 SA 文件的樣板如果定義了「需要有依賴關係 section」，而依賴關係要用什麼 diagram 呈現，是格式問題還是 rules 問題？

這不是說這個分法有錯，而是實際拆的時候會遇到邊界模糊的 case，你可能需要一個判斷基準來處理灰色地帶。一個可能的基準：「如果換一個團隊用，這個東西會不會改變？」會改 → caller 管；不會改 → 留在 skill。

## 使用者回應：彈性與穩定的分層

- 關於 silent failure，我認為從使用 SKILL 和 LLM 開始，就已經會有這個潛在問題，需要從「彈性」、「穩定」中找平衡，沒有辦法完美。我的想法中，caller 是一個能夠需要決定要「穩定」的端點，但底層的 flow 或邏輯就要相對的「彈性」，才有「疊加規則」、「疊加規範」的空間。如果一開始 analyst-system 直接制定了一定要遵守哪些規範，一定要有哪些產出，一定要用哪種方式，那麼就會讓這個底層的 flow 變得「穩定」，但失去了調整的彈性。我認為透過「疊加規則、規範」來達成穩定，是一個可行的方向。

- 如果制定了「樣板」，代表我添加了某種「規範」，讓指標偏向「穩定」，那這份「穩定」我就希望只用者遵守。另外，換一個團隊，caller 也能夠透過上下文的指定樣板。或者，有可能樣板不留在 Skill，而是再包裝一層。

你有其他看法嗎？

---

## Claude 最終回應

你把「穩定」和「彈性」分層的思路很清晰——底層 flow 保持彈性，穩定性由 caller 透過疊加來實現，而不是由底層自己鎖死。這個分層邏輯我沒有異議。

關於樣板，你最後那句「有可能樣板不留在 Skill，而是再包裝一層」——如果順著你前面的邏輯推到底，這其實是更一致的做法。樣板本質上就是一種規範（規定產出的結構），跟 rules 的性質相同，只是管的面向不同。把 rules 移出去但把樣板留下來，邏輯上有點不對稱。

所以如果追求一致性，analyst-system 就是純粹的流程定義——做什麼、什麼順序、什麼條件分支——其他全部外部注入。樣板由 caller 或中間層提供。

但這裡有一個實際的取捨：拆得越乾淨，caller 要準備的東西越多，寫一個新 caller 的門檻就越高。如果你預期 caller 數量不多且都是你自己寫，這不是問題。但如果未來有其他人要包裝，他需要同時準備 rules、目錄結構、樣板，缺一個就可能 silent failure。

這不是反對你的方向，只是指出：一致性和上手成本之間有個 tradeoff，你可以根據實際的使用情境決定要多乾淨。

## 核心概念摘要

### 分層架構

```
使用者（自然語言 context / 額外規範）
  ↓ 參數
caller（executing-sa command）— 穩定端：注入 rules、目錄結構、樣板
  ↓ 調用
analyst-system（純 workflow）— 彈性端：只定義流程
```

### 穩定與彈性的關係

- **底層（analyst-system）**：保持彈性，不內建任何外部依賴，為「疊加規則」留空間
- **caller 端**：透過疊加 rules、規範、目錄結構來達成穩定
- **穩定不是由底層鎖死，而是由 caller 透過組合來實現**

### 關鍵 Tradeoff

| 面向 | 取捨 |
|------|------|
| 彈性 vs 穩定 | 底層彈性，caller 端穩定；接受 silent failure 風險 |
| 一致性 vs 上手成本 | 拆得越乾淨，新 caller 門檻越高 |
| 樣板歸屬 | 暫留 skill（務實），但邏輯上應移出（一致性）|
| 前置條件 | 不加在流程中，需要的話寫使用文件 |
