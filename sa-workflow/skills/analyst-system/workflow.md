# SA Workflow 規範

## 前提
- 已完成 RA 階段，使用者持有確認過的規格文件（以下稱「新規格」）。
- 新規格由使用者提供。

---

## 產出物目錄結構

所有產出文件放置在執行專案的根目錄下，依以下結構組織：

```
{project-root}/
├── .sa/
│   ├── codebase-analyze/        # 步驟一：既有 Codebase 規格文件
│   │                            # 步驟二：既有 Codebase 架構文件
│   ├── 新舊規格差異比對.md        # 步驟四：新舊規格差異文件
│   ├── SA文件.md                 # 步驟六：SA 文件
│   └── design/                  # 步驟八：Design 文件
└── .tasks/                      # 步驟十：Task 文件
```

- Task 文件內引用 SA 文件、Design 文件時，使用從專案根目錄起算的路徑（例如 `.sa/design/xxx.md`）。

### 產出物交付規則

- 所有步驟的產出物一律寫成檔案，放置於上述對應目錄。
- 若產出內容存在不確定或多重解讀的部分，在文件內直接標記。
- Human-in-loop 步驟：除了寫檔外，在對話中提供檔案連結與摘要，供使用者確認。

---

## 階段零：初始化

進入 workflow 前，執行兩輪資訊收集，決定本次的執行路徑。

### 第一輪：收集輸入物

向使用者詢問並確認以下項目：

1. **新規格**（必要）：使用者提供的新規格文件。
2. **既有 Codebase**：是否有需要分析的既有線上代碼？
   - 若有，追加詢問：
     - Codebase 的入口檔案或目錄
     - 與新規格相關的模組範圍（若使用者能提供）
3. **既有規格文件**：是否已有既有 codebase 的規格文件？
4. **既有架構文件**：是否已有既有 codebase 的架構文件（ClassDiagram、SequenceDiagram）？

根據使用者回應，自動決定以下步驟的執行狀態：

| 條件 | 影響 |
|------|------|
| 無既有 codebase | 跳過步驟一、二、三、四、五，步驟六使用六-B |
| 有既有 codebase，但無既有規格文件 | 執行步驟一 |
| 有既有 codebase，且有既有規格文件 | 跳過步驟一 |
| 有既有 codebase，但無既有架構文件 | 執行步驟二 |
| 有既有 codebase，且有既有架構文件 | 跳過步驟二 |
| 有既有 codebase | 步驟六使用六-A |

### 第二輪：確認 Human-in-loop 偏好

將以下 optional 的 Human-in-loop 步驟列出，詢問使用者哪些要保留：

- **步驟三**：確認既有 codebase 的規格/架構文件
- **步驟五**：確認新舊規格差異
- **步驟九**：確認 Design 文件

說明：步驟七（確認 SA 文件）為必要步驟，不可跳過。

### 初始化完成

將本次執行路徑整理後呈現給使用者確認，格式如下：

```
本次 SA Workflow 執行路徑：
- 步驟一：[執行 / 跳過]（原因）
- 步驟二：[執行 / 跳過]（原因）
- 步驟三：[執行 / 跳過]
- 步驟四：[執行 / 跳過]
- 步驟五：[執行 / 跳過]
- 步驟六：[六-A / 六-B]
- 步驟七：執行（必要）
- 步驟八：執行
- 步驟九：[執行 / 跳過]
- 步驟十：執行
```

使用者確認後，開始執行。

## Human-in-loop 通用規則

適用於所有標記為 Human-in-loop 的步驟。

1. 將該步驟的產出物完整呈現給使用者。
2. 若產出物中有標記不確定或多重解讀的部分，在摘要中一併列出，引導使用者重點確認。
3. 詢問使用者：
   - 內容是否正確？
   - 是否有需要修正的部分？
3. 等待使用者回應，不可自行往下執行。
4. 根據使用者回應：
   - **確認通過** → 進入下一步驟。
   - **提出修正** → 根據修正內容更新產出物，重新呈現，再次詢問確認。重複此循環直到使用者確認通過。

---

## 步驟一：既有 Codebase 規格分析

**條件**：有既有 codebase，且使用者未提供既有規格文件時執行。

分析既有 codebase，撰寫規格文件，描述目前系統的功能與行為。

**產出**：既有 codebase 規格文件。

---

## 步驟二：既有 Codebase 架構分析

**條件**：有既有 codebase，且使用者未提供既有架構文件時執行。

分析既有 codebase，繪製架構文件，包含：
- ClassDiagram
- SequenceDiagram

**產出**：既有 codebase 架構文件（ClassDiagram、SequenceDiagram）。

---

## 步驟三：確認規格/架構文件（Human-in-loop・optional）

**條件**：初始化階段使用者選擇保留此步驟時執行。

依照 Human-in-loop 通用規則，請使用者確認本次實際執行的產出物：
- 若步驟一有執行 → 確認既有 codebase 規格文件。
- 若步驟二有執行 → 確認既有 codebase 架構文件。
- 若步驟一和步驟二皆有執行 → 兩份產出物都需確認。

---

## 步驟四：新舊規格差異比對

**條件**：有既有 codebase 時執行。

將既有規格（步驟一產出或使用者提供）與新規格進行比對，產出差異文件。

**產出**：新舊規格差異文件。

---

## 步驟五：確認新舊規格差異（Human-in-loop・optional）

**條件**：初始化階段使用者選擇保留此步驟時執行。

依照 Human-in-loop 通用規則，請使用者確認步驟四的產出物。

---

## 步驟六：撰寫 SA 文件

根據初始化階段的判斷，執行六-A 或六-B。

### 六-A：有 Codebase

針對新舊規格差異與既有 codebase，進行以下工作：
1. 依照功能進行分類。
2. 拆解出每一個需要異動的組件/模組。
3. 寫出每一個檔案位置與功能之間的依賴關係。

### 六-B：無 Codebase

根據新規格，進行以下工作：
1. 依照功能進行分類。
2. 標記每個功能之間的依賴關係。

**產出**：SA 文件。

---

## 步驟七：確認 SA 文件（Human-in-loop・必要）

此步驟不可跳過。

依照 Human-in-loop 通用規則，請使用者確認步驟六的產出物。

---

## 產出 Design 文件（步驟八 + 步驟九 循環）

步驟八和步驟九組成一個循環，每次只針對一個功能進行 Design。
完成一個功能的 Design 並確認後，再進行下一個功能。不平行處理。

循環順序依照 SA 文件中的功能分類，逐一執行。

---

## 步驟八：產出 Design 文件

針對當前功能，進行架構設計，包含：
- 參考既有 codebase 的設計（若有）
- 依照新規格設計架構
- 繪製 ClassDiagram
- 繪製 SequenceDiagram

**產出**：該功能的 Design 文件。

---

## 步驟九：確認 Design 文件（Human-in-loop・optional）

**條件**：初始化階段使用者選擇保留此步驟時執行。

依照 Human-in-loop 通用規則，請使用者確認步驟八的產出物。

確認通過後，若 SA 文件中仍有未處理的功能，回到步驟八處理下一個功能。
所有功能的 Design 皆完成後，進入步驟十。

---

## 步驟十：依照 Design 文件產出 Task

根據所有 Design 文件、SA 文件、規格文件的內容，將實作工作拆解為具體的 task。

拆解方式由 LLM 根據功能依賴關係、架構層次等因素綜合判斷，選擇合適的拆分策略。

### Task 產出要求

- 遵循 Task 文件規範。
- 每份 Task 須達到「摘要自足」——開發者單獨看 Task 就能理解做什麼、怎麼做、為什麼。
- 引用 SA 文件、Design 文件的路徑作為細節參考，而非前置必讀。

**產出**：Task 清單。
