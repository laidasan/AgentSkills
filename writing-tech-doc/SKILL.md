# writing-tech-doc

將對話中的技術知識討論，經過提取、分類、對齊，最終產出結構化的 Markdown 技術文件。

## 為什麼需要這個流程

每次將討論轉為文件時，都要重新對齊章節結構和篇幅深度，這個流程透過樣板機制固定骨架，把精力留給內容本身。同時保留討論階段的彈性，讓使用者能調整篇幅與深度。

## 流程

嚴格按照以下五個步驟循序執行，不可跳過任何步驟。

### 步驟 1：提取知識點

分析當前對話內容，提取討論過的技術知識點。

- 只提取技術相關內容，過濾一般對話（寒暄、確認、離題）
- 產出一份知識點清單，每個知識點用一句話概括
- 將清單呈現給使用者確認，確保沒有遺漏或誤判

### 步驟 2：判斷主題類型與選擇樣板

根據知識點的性質，判斷主題屬於哪種類型，並讀取對應的樣板檔。

**兩種類型的判斷依據：**

| 類型 | 特徵 | 樣板 |
|---|---|---|
| 取代型 | 討論中存在「舊方案 vs 新方案」的對比關係，新技術是為了取代或改善既有做法 | `templates/template-replacement.md` |
| 獨立概念型 | 討論的是一個獨立的技術概念，沒有明確的新舊替換關係 | `templates/template-standalone.md` |

讀取對應的樣板檔案，取得文件的章節骨架。

如果無法明確歸類，向使用者說明判斷依據，由使用者決定採用哪個樣板。

### 步驟 3：產出大綱

依據樣板的章節結構，將步驟 1 提取的知識點分配到各章節，產出文件大綱。

大綱格式：
- 列出每個章節的標題
- 每個章節下列出預計涵蓋的要點（來自知識點清單）
- 標注哪些章節的內容在對話中討論較深、哪些較淺

### 步驟 4：討論對齊

將大綱呈現給使用者，進行討論。這個階段的自由度高，根據情境判斷需要討論什麼，可能包括：

- 篇幅深度（哪些章節需要更詳細、哪些可以精簡）
- 章節調整（增刪、合併、拆分、重新排序）
- 內容補充（對話中未深入但文件中需要涵蓋的部分）
- 程式碼範例的取捨與呈現方式

不限來回次數，直到使用者明確表示確認為止。

**進入下一步的條件：使用者明確表示確認（例如「OK」、「可以了」、「就這樣」）。** 不要在使用者尚未確認時就開始撰寫。

### 步驟 5：撰寫文件

根據確認後的大綱，結合對話中的討論內容，撰寫完整的 Markdown 文件並寫入檔案。

- 輸出語言：根據對話語言和技術內容的慣例判斷
- 檔名與路徑：根據主題內容和當前工作目錄判斷合適的命名與位置
- 撰寫時以對話中實際討論的內容為主要素材，不額外發明對話中未提及的內容

## 例外處理

- **對話中無明確技術知識點：** 告知使用者，說明在對話中未找到足夠的技術知識點來產出文件，不強制產出。
