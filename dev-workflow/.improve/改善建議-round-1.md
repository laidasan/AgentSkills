# 改善建議 — Round 1

## 測試素材摘要

- **素材來源：** `.test-doc/.tasks/bgaent-feedback/` 下的 Task 文件，搭配 `.test-doc/` 下的 Design 文件
- **任務類型：** 新功能開發（回饋收集與累計機制 + 情境判斷引擎）
- **複雜度：** 中高 — 多個 class 間有依賴關係，業務邏輯有互斥與觸發規則
- **是否有既有 codebase：** 有（Vue 2 + 自建 Manager 架構）

---

## 執行過程中發現的問題

### 1. Task 路徑假設過於固定

**問題：** 步驟一寫死搜尋 `.tasks/` 目錄，但實際 task 放在 `.test-doc/.tasks/bgaent-feedback/`。使用者需要手動指定 task 路徑。

**影響：** 步驟一直接失敗（找不到 task），流程中斷。

**建議：** `.tasks/` 作為預設路徑，但允許使用者指定 task 路徑或 task 文件。步驟一改為：先搜尋 `.tasks/`，若找不到，詢問使用者 task 來源。

---

### 2. Task 依賴關係未被流程處理

**問題：** Task 04 標注「依賴：Task 02」，但 workflow 步驟一～二只關注「有哪些 task 要做」，沒有處理依賴排序。實際執行時是我主動發現依賴問題後詢問使用者。

**影響：** 如果直接執行 Task 04 而忽略依賴，會因為缺少 `FeedbackAccumulatorManager` 而卡住。

**建議：** 在步驟一之後、步驟二之前，加入依賴解析步驟：
- 讀取所有 task 的依賴欄位
- 若選定的 task 有未完成的依賴，提示使用者並建議先執行依賴 task
- 或自動將依賴 task 加入執行佇列並排在前面

---

### 3. Task 的 Design 文件關聯不明確

**問題：** Task 04 有對應的 Design 文件（`步驟八-Design-03-情境判斷引擎.md`），但 task 文件本身沒有標注 Design 文件路徑。是使用者主動告知要一起載入。

**影響：** 如果使用者沒有提醒，可能漏讀 Design 文件，導致實作與設計不一致。

**建議：** 兩個方向（非互斥）：
1. Task 文件中增加 `相關文件` 欄位，明確列出 Design 等參考文件
2. 步驟五開始執行前，主動搜尋與 task 名稱相關的文件（Design、SA、規格等）

---

### 4. 步驟三的「現有 Context」來源過於模糊

**問題：** 步驟三要從「現有 LLM 上下文」尋找開發規範，但 workflow 開始時上下文幾乎是空的。實際做法是主動去翻 codebase 的檔案結構、既有程式風格、測試慣例等。

**影響：** 「從現有 Context 尋找」和「主動去 codebase 搜尋」是不同的行為，目前步驟三的描述讓人不確定該做到什麼程度。

**建議：** 步驟三改為明確的兩階段：
1. 檢查已載入的 context（`.claude/rules/`、對話中提供的規範、CLAUDE.md 等）
2. 主動從 codebase 推斷開發慣例（檔案結構、命名風格、import 慣例、測試框架）

---

### 5. 環境相容性問題未被流程涵蓋

**問題：** 跑測試時遇到 OpenSSL 相容性錯誤（`ERR_OSSL_EVP_UNSUPPORTED`），需要加 `NODE_OPTIONS=--openssl-legacy-provider`。另外 `??` 語法不被專案 ESLint parser 支援。

**影響：** 這兩個問題各浪費了一次執行迴圈。

**建議：** 在步驟三（確認開發規範）時，增加一個子項：確認環境限制，包括：
- 測試指令是否能正常執行（先跑一次空測試）
- 語法限制（ESLint/babel 設定支援的語法範圍）

---

### 6. 步驟五～七 Loop 內缺少「載入相關文件」的動作

**問題：** 步驟五直接說「依照開發規範執行 task」，但在執行前還需要載入 task 相關的 Design 文件、既有程式碼等。目前這個動作是隱含的，沒有被流程明確描述。

**建議：** 在步驟五之前（或作為步驟五的第一個子步驟），加入：
- 載入該 task 的所有相關文件（Design、參考程式碼、依賴 class 等）
- 確認所有必要資訊已具備後再開始實作

---

## 流程面回顧

### 順暢的部分
- 步驟六（規範檢查）運作良好 — 有明確的驗收條件和 TODO 列表，逐項對照很清楚
- 步驟七（標記完成）簡單直覺
- 步驟五～七的 Loop 結構在連續執行兩個 task（02 → 04）時運作順暢
- 依賴 task 先做再做目標 task 的決策流程自然

### 可改進的部分
- 步驟一～二偏靜態，沒有處理 task 之間的關係
- 步驟三的範圍模糊，實際執行時做了很多步驟三沒有描述的事
- 缺少「載入相關文件」的明確步驟

---

## 建議的優先順序

1. **【高】** Task 依賴解析 — 不處理會導致執行順序錯誤
2. **【高】** Task 相關文件關聯 — 不處理會導致漏讀重要設計文件
3. **【中】** 步驟三細化 — 改善可操作性
4. **【中】** Task 路徑彈性化 — 提高適用範圍
5. **【低】** 環境檢查 — 屬於效率優化
6. **【低】** 載入相關文件步驟 — 明確化既有隱含行為
