# 1. 文件資訊區

## 目的

提供文件版本追蹤和異動歷史，確保所有人都能掌握最新變更。

## 撰寫規範

### 基本格式

```markdown
# [專案名稱] PRD

**版本**：v20250203
**最後更新**：2025/02/03
**負責 PM**：[姓名]
**相關人員**：[工程師姓名]、[設計師姓名]

## 變更日誌（近期異動）
> 此區塊僅保留近三天的異動，每週或每個 Sprint 結束後歸檔至修訂歷史

| 日期 | 受影響模組 | 異動內容 | 原因 | 錨點連結 |
|------|-----------|---------|------|---------|
| 02/03 | 購物車 | 調整 API 欄位從 ID 改為 UUID | 配合後端架構 | [跳轉](#購物車功能) |
| 02/02 | 付款功能 | 移除「貨到付款」選項 | 時程壓力，延後至 v2.0 | [跳轉](#付款功能) |

## 修訂歷史
| 版本 | 日期 | 異動摘要 | 修訂人 |
|------|------|---------|--------|
| v20250203 | 2025/02/03 | 調整購物車 API 規格 | [PM 姓名] |
| v20250201 | 2025/02/01 | 初版完成 | [PM 姓名] |
```

## 關鍵要素

### 1. 版本號格式
- **格式**：`vYYYYMMDD`（時間戳記格式）
- **範例**：v20250203 代表 2025 年 2 月 3 日版本
- **更新時機**：每次重大變更時更新版本號

### 2. 變更日誌
- **保留範圍**：僅保留近 3 天的異動
- **歸檔時機**：每週或每個 Sprint 結束後歸檔至修訂歷史
- **必要欄位**：
  - 日期：異動日期
  - 受影響模組：哪個功能或章節受到影響
  - 異動內容：做了什麼改變
  - 原因：為什麼要改變
  - 錨點連結：快速跳轉到相關章節（使用 Markdown 錨點）

### 3. 修訂歷史
- **用途**：歸檔所有歷史版本的重大變更
- **記錄內容**：版本號、日期、異動摘要、修訂人
- **排序**：由新到舊排列

## 錨點連結設定方式

在 Markdown 中，使用以下格式建立錨點連結：

```markdown
<!-- 目標章節 -->
## 購物車功能

<!-- 連結到該章節 -->
[跳轉](#購物車功能)
```

對於包含空格的標題，錨點會自動轉換為小寫並將空格替換為連字號。

## 注意事項

1. **版本號一致性**：文件開頭的版本號和修訂歷史中的最新版本必須一致
2. **異動內容要具體**：避免「調整功能」、「修改規格」等模糊描述，要明確說明改了什麼
3. **原因要說明背景**：說明為什麼要改，常見原因包括：
   - 配合後端架構調整
   - 時程壓力，功能延後
   - 業務需求變更
   - 技術可行性評估後調整
4. **錨點連結要有效**：確保連結能正確跳轉到對應章節
5. **定期歸檔**：避免變更日誌過長，影響閱讀體驗

## 常見錯誤

❌ **錯誤示範**：
```markdown
| 02/03 | 功能 | 修改 | 調整 | [連結](#) |
```
**問題**：資訊不明確，無法理解具體改了什麼

✅ **正確示範**：
```markdown
| 02/03 | 購物車 | 調整 API 欄位從 ID 改為 UUID | 配合後端架構 | [跳轉](#購物車功能) |
```
**優點**：清楚說明改了什麼、為什麼改、如何查看

## 適用場景

- ✅ 專案啟動時建立初版文件資訊區
- ✅ 每次需求變更時更新變更日誌
- ✅ Sprint 結束時歸檔變更日誌到修訂歷史
- ✅ 文件交接時，讓新成員快速了解專案演進

## 與其他章節的關聯

- **名詞定義**：變更日誌中提到的術語應在名詞定義中有明確定義
- **功能詳述**：錨點連結應指向功能詳述中的具體功能章節
- **優先級**：若因時程壓力調整功能，應在優先級章節更新對應的優先級標註
