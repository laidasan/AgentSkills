# PRD 品質檢查清單

## 使用說明

此檢查清單用於驗證 PRD 的完整性、一致性、清晰度和可執行性。建議在 PRD 完成後逐項檢查。

---

## 1. 完整性檢查

### 文件結構
- [ ] 文件資訊區已完成（版本號、變更日誌、修訂歷史）
- [ ] 名詞定義已完成（業務術語、技術術語）
- [ ] 問題定義與目標已完成（問題描述、目標使用者、預期目標）
- [ ] 使用者故事與情境已完成（至少 2-3 個情境）
- [ ] 功能範疇與邊界已完成（本次包含、本次不包含、已知限制）
- [ ] 外部依賴與技術約束已完成（後端服務、第三方服務、開發順序）
- [ ] 互動流程與使用者行為已完成（主要流程、替代流程、異常流程）
- [ ] 功能詳述已完成（所有功能都有描述、前置條件、驗收標準）
- [ ] 資料與規則定義已完成（資料結構、欄位說明、業務規則）
- [ ] 邊界條件與異常處理已完成（資料邊界、權限處理、系統異常、操作衝突）
- [ ] 優先級已完成（所有功能都有優先級標註）

### 功能詳述
- [ ] 每個功能都有功能描述
- [ ] 每個功能都有前置條件
- [ ] 每個功能都有驗收標準（Given/When/Then 格式）
- [ ] 驗收標準涵蓋正常、邊界、異常情境
- [ ] 設計稿連結已附上（若已有設計稿）

### 資料定義
- [ ] 所有關鍵資料結構都有定義
- [ ] 所有欄位都有型別、必填、驗證規則、說明
- [ ] 業務規則已明確說明（數量限制、時間限制、計算公式）

---

## 2. 一致性檢查

### 術語一致性
- [ ] 名詞使用前後一致（檢查是否有同義詞混用）
- [ ] 資料欄位名稱與 API 文件一致（若有 API 文件）
- [ ] 驗收標準中的術語與名詞定義一致
- [ ] UI 提示文字中的術語與名詞定義一致

### 邏輯一致性
- [ ] 驗收標準與功能描述一致
- [ ] 互動流程與驗收標準一致
- [ ] 邊界條件與驗收標準一致
- [ ] 優先級與問題定義的目標對應

### 數據一致性
- [ ] 數量限制前後一致（如購物車上限在不同章節的描述一致）
- [ ] 時間限制前後一致（如逾時時間、保留期限）
- [ ] 計算公式前後一致

---

## 3. 清晰度檢查

### 術語定義
- [ ] 所有關鍵術語都有明確定義
- [ ] 術語定義具體且無歧義
- [ ] 術語定義避免循環定義

### 驗收標準
- [ ] 驗收標準使用 Given/When/Then 格式
- [ ] Given 描述了完整的前提條件
- [ ] When 描述了明確的觸發動作
- [ ] Then 描述了具體且可驗證的預期結果

### 異常處理
- [ ] 異常處理有明確的系統行為
- [ ] 異常處理有明確的 UI 顯示
- [ ] UI 提示文字具體且友善（避免技術術語）

### 優先級
- [ ] 優先級調整有說明原因
- [ ] 調整原因具體且清楚（避免「重要性較低」等模糊描述）

---

## 4. 可執行性檢查

### 工程師視角
- [ ] 工程師能根據 PRD 開始開發
- [ ] 資料結構定義完整且格式正確
- [ ] 驗證規則明確且可實作
- [ ] 異常處理邏輯清楚
- [ ] API 文件連結已提供（若有）

### 設計師視角
- [ ] 設計師能根據 PRD 設計 UI
- [ ] 互動流程清楚且完整
- [ ] UI 提示文字已定義
- [ ] 使用者故事提供了足夠的脈絡

### 測試人員視角
- [ ] 測試人員能根據驗收標準撰寫測試案例
- [ ] 驗收標準具體且可測試
- [ ] 邊界條件和異常情境已涵蓋
- [ ] 測試數據範例已提供（若需要）

---

## 5. 風險管理檢查

### 外部依賴
- [ ] 所有外部依賴都已列出
- [ ] 每個依賴項目都有風險等級標註（🟢/🟡/🔴）
- [ ] 中高風險項目都有備援方案
- [ ] 備援方案具體且可執行（避免「再討論」、「待確認」）
- [ ] 預計完成日已與負責人確認

### 技術約束
- [ ] 技術約束已明確列出
- [ ] 已知限制已說明原因
- [ ] 開發順序約束已標註

---

## 6. 脈絡檢查

### 問題定義
- [ ] 問題描述具體且有數據支撐
- [ ] 目標使用者明確且有主次之分
- [ ] 預期目標可衡量（有具體指標）
- [ ] 預期目標與問題描述對應

### 使用者故事
- [ ] 每個情境都有角色、情境、使用前、使用後四要素
- [ ] 情境具體且有代入感
- [ ] 至少涵蓋 2-3 個主要情境
- [ ] 情境與目標使用者對應

### 功能範疇
- [ ] 功能範疇與問題定義的痛點對應
- [ ] 「本次不包含」的功能有說明原因
- [ ] 已知限制有區分技術限制和業務限制

---

## 7. 邊界處理檢查

### 資料邊界
- [ ] 涵蓋空資料情況（如購物車為空）
- [ ] 涵蓋數量上限情況（如購物車滿、商品數量達上限）
- [ ] 涵蓋異常輸入情況（如非數字、負數、超大數字）
- [ ] 涵蓋長度限制情況（如商品名稱過長）

### 系統異常
- [ ] 涵蓋網路異常情況（如斷線、逾時）
- [ ] 涵蓋 API 錯誤情況（如 500、503）
- [ ] 涵蓋儲存空間異常情況（如 localStorage 滿）

### 權限處理
- [ ] 涵蓋未登入情況
- [ ] 涵蓋 Session 過期情況
- [ ] 涵蓋權限不足情況

### 操作衝突
- [ ] 涵蓋並發操作情況（如多裝置同時修改）
- [ ] 涵蓋資料競爭情況（如多人搶購）
- [ ] 有明確的衝突處理策略

---

## 8. 文件品質檢查

### 可讀性
- [ ] 使用標題、列表、表格等結構化格式
- [ ] 使用符號增加可讀性（如 ✅/❌、🟢/🟡/🔴）
- [ ] 段落長度適中，不過於冗長
- [ ] 使用範例和圖表輔助說明

### 可維護性
- [ ] 變更日誌有錨點連結，方便快速定位
- [ ] 章節結構清晰，便於查找
- [ ] 版本號格式一致（vYYYYMMDD）

### 可追溯性
- [ ] 變更日誌記錄了異動原因
- [ ] 優先級調整有說明原因
- [ ] 功能範疇變更有記錄

---

## 9. 協作檢查

### 跨團隊溝通
- [ ] PM、工程師、設計師都能理解 PRD
- [ ] 專業術語都有定義，避免誤解
- [ ] 設計稿連結有效且可開啟

### 決策記錄
- [ ] 重要決策有記錄（如為什麼不做某功能）
- [ ] 優先級調整有說明原因
- [ ] 備援方案的選擇有說明理由

---

## 10. 最終檢查

### 整體檢視
- [ ] 通讀整份 PRD，確認邏輯連貫
- [ ] 確認沒有遺漏的章節或功能
- [ ] 確認沒有相互矛盾的描述

### 團隊共識
- [ ] PRD 已與工程團隊討論並達成共識
- [ ] PRD 已與設計團隊討論並達成共識
- [ ] 利益相關人已審閱並同意

### 版本管理
- [ ] 版本號已更新
- [ ] 變更日誌已更新
- [ ] 修訂歷史已更新

## 附註

### 檢查頻率建議
- **專案啟動時**：完整檢查所有項目
- **每週檢視**：檢查變更日誌、優先級、外部依賴
- **Sprint 結束時**：檢查完整性、一致性、可執行性
- **重大變更時**：重新檢查受影響的章節

### 檢查工具
- **自動化檢查**：使用 Linter 檢查 Markdown 格式
- **協作檢查**：使用 Figma/Miro 進行團隊 Review
- **版本管理**：使用 Git 追蹤 PRD 變更歷史

### 常見問題
**Q：檢查清單太長，需要全部檢查嗎？**
A：建議至少完成「完整性檢查」、「一致性檢查」、「可執行性檢查」三大類。其他項目可根據專案複雜度選擇性檢查。

**Q：誰應該執行這個檢查？**
A：建議 PM 先自檢，然後與工程師、設計師一起 Review，確保所有人都理解且同意 PRD 內容。

**Q：檢查發現問題怎麼辦？**
A：記錄問題並逐項修正，修正後更新版本號和變更日誌，並通知相關人員。
