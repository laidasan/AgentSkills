# 2. 名詞定義

## 目的

集中定義所有關鍵術語，確保 PM、工程師、設計師對專案術語有一致理解。

## 撰寫規範

### 基本結構

將術語分為「業務術語」和「技術術語」兩大類：

#### 業務術語
```markdown
## 名詞定義

### 業務術語

| 術語 | 定義 | 備註 |
|------|------|------|
| 購物車 | 使用者暫存欲購買商品的清單，支援新增、刪除、修改數量 | - |
| 有效商品 | 庫存 > 0 且狀態為「上架中」的商品 | 下架商品會從購物車自動移除 |
| 訪客 | 未登入的使用者 | 訪客的購物車資料存於瀏覽器 localStorage |
| 會員 | 已登入的使用者 | 會員的購物車資料存於後端資料庫 |
| 小計 | 單一商品的總價，計算公式：單價 × 數量 | - |
| 總金額 | 購物車內所有商品的小計加總 | 不含運費、折扣等 |
```

#### 技術術語
```markdown
### 技術術語

| 術語 | 定義 | 備註 |
|------|------|------|
| UUID | Universally Unique Identifier，全域唯一識別碼 | 用於購物車 ID、商品 ID 等 |
| localStorage | 瀏覽器本地儲存空間，容量約 5-10MB | 訪客購物車資料存於此 |
| Session | 使用者登入後的會話狀態，通常有時效性 | 會員登入後會建立 Session |
| API Timeout | API 請求逾時時間，本專案設定為 5 秒 | 超過 5 秒視為網路異常 |
| CRUD | Create、Read、Update、Delete，資料的增刪改查操作 | 購物車 API 提供完整 CRUD 功能 |
```

## 關鍵要素

### 1. 分類原則
- **業務術語**：主要給 PM、設計師閱讀，描述業務邏輯和使用者行為
- **技術術語**：主要給工程師閱讀，描述技術實作和系統行為
- **跨領域術語**：若某術語橫跨兩者（如「購物車 ID」），可在業務術語中定義基本概念，技術術語中補充技術細節

### 2. 定義撰寫原則
- **明確性**：定義要精確，避免模糊或循環定義
- **完整性**：包含範圍、條件、計算公式等關鍵資訊
- **一致性**：與整份 PRD 使用的術語保持一致

### 3. 備註欄位
用於補充說明：
- 使用場景
- 特殊規則
- 與其他術語的關係
- 技術實作細節

## 排序建議

### 方式一：依使用頻率排序
將最常用的術語放在最前面，方便快速查閱。

### 方式二：依字母順序排序
適合術語數量較多時，便於搜尋。

### 方式三：依邏輯分組
將相關術語放在一起，例如：
- 使用者相關：訪客、會員、管理員
- 商品相關：商品、庫存、價格
- 交易相關：購物車、訂單、付款

**建議**：優先使用「依邏輯分組」，分組內再依使用頻率或字母排序。

## 注意事項

1. **避免縮寫**：首次出現時應寫全名，並標註縮寫（例如：Product Requirement Document, PRD）
2. **避免循環定義**：術語 A 的定義不應依賴術語 B，而術語 B 又依賴術語 A
3. **保持同步更新**：當 PRD 其他章節新增術語時，需同步更新此章節
4. **明確數值範圍**：若術語涉及數字（如「庫存」、「數量」），應明確範圍（如 ≥ 0）
5. **說明特殊狀態**：若術語有特殊狀態或邊界情況，應在定義或備註中說明

## 常見錯誤

❌ **錯誤示範 1**：定義不明確
```markdown
| 購物車 | 使用者購物的地方 | - |
```
**問題**：「購物的地方」太模糊，未說明購物車的功能和作用

✅ **正確示範**：
```markdown
| 購物車 | 使用者暫存欲購買商品的清單，支援新增、刪除、修改數量 | - |
```

❌ **錯誤示範 2**：循環定義
```markdown
| 訂單 | 包含購物車內容的交易記錄 | - |
| 購物車 | 訂單前的商品暫存清單 | - |
```
**問題**：兩者互相依賴，定義不清楚

✅ **正確示範**：
```markdown
| 購物車 | 使用者暫存欲購買商品的清單，支援新增、刪除、修改數量 | - |
| 訂單 | 使用者提交購物車後產生的交易記錄，包含商品明細、金額、收貨資訊等 | - |
```

❌ **錯誤示範 3**：業務術語和技術術語混雜
```markdown
| 購物車 | 使用者暫存商品的清單，使用 localStorage 儲存 | - |
```
**問題**：混雜了業務概念和技術細節

✅ **正確示範**：
```markdown
### 業務術語
| 購物車 | 使用者暫存欲購買商品的清單 | - |

### 技術術語
| localStorage | 瀏覽器本地儲存空間 | 訪客購物車資料存於此 |
```

## 適用場景

- ✅ 專案啟動時，收集並定義所有核心術語
- ✅ 需求討論時，發現有歧義的術語時立即定義
- ✅ 跨團隊協作時，確保所有人對術語理解一致
- ✅ 新成員加入時，快速理解專案術語

## 與其他章節的關聯

- **變更日誌**：變更日誌中提到的術語應在此章節有定義
- **功能詳述**：功能描述中使用的術語應與此章節定義一致
- **資料與規則定義**：資料欄位名稱應與此章節的術語對應
- **驗收標準**：驗收標準中的術語應與此章節定義一致

## 維護建議

1. **定期審查**：每個 Sprint 結束後，審查術語定義是否需要更新
2. **版本管理**：若術語定義有重大變更，應在變更日誌中記錄
3. **團隊共識**：重要術語的定義應經過團隊討論達成共識
4. **避免過度定義**：不需要定義所有術語，只定義「關鍵」且「可能有歧義」的術語
