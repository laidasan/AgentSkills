# 3. 問題定義與目標

## 目的

說明為什麼要做這個專案，要解決什麼問題，讓所有人理解專案的價值和方向。

## 撰寫規範

### 基本結構

```markdown
## 問題定義與目標

### 問題描述
目前使用者在選購商品時，無法暫存欲購買的商品清單，導致：
- 每次需重新搜尋商品，造成操作不便
- 使用者放棄購買的比例較高（約 40%）
- 客服接到大量「如何儲存商品」的詢問

### 目標使用者
- 主要：瀏覽商品但尚未決定購買的使用者
- 次要：需要分批採購的企業採購人員

### 預期目標
1. 降低使用者放棄購買的比例（目標：降至 25%）
2. 提升回購率（目標：提升 15%）
3. 減少客服關於「如何儲存商品」的詢問量（目標：降低 50%）
```

## 關鍵要素

### 1. 問題描述
- **具體性**：避免「體驗不好」、「效率不高」等模糊描述
- **數據支撐**：盡量附上具體數據或比例
- **影響範圍**：說明問題影響哪些使用者、哪些場景
- **多維度呈現**：可從使用者痛點、業務損失、成本增加等角度描述

#### 問題描述的結構建議
```markdown
目前 [使用者/角色] 在 [場景] 時，[遇到什麼問題]，導致：
- [影響 1]（附數據）
- [影響 2]（附數據）
- [影響 3]（附數據）
```

### 2. 目標使用者
- **主次分明**：區分主要使用者和次要使用者
- **具體描述**：避免「所有使用者」等過於寬泛的描述
- **場景關聯**：說明使用者在什麼場景下會使用此功能

#### 常見使用者分類
- 依角色：訪客、會員、管理員
- 依行為：瀏覽型、購買型、比價型
- 依需求：個人消費、企業採購、批量購買

### 3. 預期目標
- **可衡量性**：每個目標都應有明確的衡量指標
- **合理性**：目標應基於數據分析和業務經驗設定
- **時間範圍**：可標註預計達成時間（如：上線後 3 個月內）
- **優先順序**：若有多個目標，可標註優先順序

#### 目標的 SMART 原則
- **S**pecific（具體）：目標明確，不模糊
- **M**easurable（可衡量）：有數據或指標可追蹤
- **A**chievable（可達成）：目標合理，不過於理想化
- **R**elevant（相關）：與問題和專案方向一致
- **T**ime-bound（有時限）：有明確的達成時間

## 撰寫技巧

### 技巧 1：使用數據說話
❌ **模糊描述**：「使用者經常放棄購買」
✅ **具體描述**：「使用者放棄購買的比例約 40%」

### 技巧 2：從多角度描述問題
不只描述使用者痛點，也可包含：
- 業務損失（如：流失率、轉換率）
- 成本增加（如：客服成本、開發成本）
- 競爭劣勢（如：競品已有此功能）

### 技巧 3：目標要可驗證
❌ **無法驗證**：「提升使用者體驗」
✅ **可驗證**：「降低使用者放棄購買的比例至 25%」

### 技巧 4：區分「問題」和「解決方案」
這個章節應聚焦在「問題」和「目標」，而非「解決方案」。

❌ **錯誤示範**：「目前沒有購物車功能」
✅ **正確示範**：「使用者無法暫存欲購買的商品清單」

## 注意事項

1. **避免預設解決方案**：問題描述不應包含「因為沒有 XX 功能」，而是描述使用者的實際痛點
2. **數據來源要可信**：數據應來自使用者研究、數據分析、客服記錄等可信來源
3. **目標要與問題對應**：預期目標應直接對應問題描述中的痛點
4. **避免目標過多**：建議 3-5 個核心目標即可，過多會分散焦點
5. **考慮可行性**：目標設定要考慮技術可行性和資源限制

## 常見錯誤

❌ **錯誤示範 1**：問題描述不具體
```markdown
### 問題描述
目前購物體驗不好，需要改善。
```
**問題**：沒有說明具體問題是什麼，為什麼體驗不好

✅ **正確示範**：
```markdown
### 問題描述
目前使用者在選購商品時，無法暫存欲購買的商品清單，導致：
- 每次需重新搜尋商品，造成操作不便
- 使用者放棄購買的比例較高（約 40%）
```

❌ **錯誤示範 2**：目標無法衡量
```markdown
### 預期目標
1. 提升使用者滿意度
2. 增加營收
```
**問題**：沒有具體指標，無法驗證是否達成

✅ **正確示範**：
```markdown
### 預期目標
1. 降低使用者放棄購買的比例（目標：降至 25%）
2. 提升回購率（目標：提升 15%）
```

❌ **錯誤示範 3**：目標使用者過於寬泛
```markdown
### 目標使用者
- 所有使用者
```
**問題**：過於寬泛，無法聚焦

✅ **正確示範**：
```markdown
### 目標使用者
- 主要：瀏覽商品但尚未決定購買的使用者
- 次要：需要分批採購的企業採購人員
```

## 適用場景

- ✅ 專案啟動時，說明為什麼要做這個專案
- ✅ 需求評審時，讓團隊理解專案價值
- ✅ 優先級排序時，幫助決策哪些功能更重要
- ✅ 專案驗收時，檢驗是否達成預期目標

## 與其他章節的關聯

- **使用者故事與情境**：目標使用者會對應到使用者故事中的角色
- **功能範疇與邊界**：功能範疇應能解決問題描述中的痛點
- **優先級**：優先級排序應基於預期目標的重要性
- **驗收標準**：功能的驗收標準應能支撐預期目標的達成

## 驗證檢查

完成此章節後，檢查以下項目：
- [ ] 問題描述具體且有數據支撐
- [ ] 目標使用者明確且有主次之分
- [ ] 預期目標可衡量且合理
- [ ] 目標與問題直接對應
- [ ] 團隊對問題和目標有共識
